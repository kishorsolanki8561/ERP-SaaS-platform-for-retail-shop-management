## How to use this package

### Day 0 — Set up the repo

```bash
# 1. Create your repo on GitHub / Azure DevOps.
git clone <your-repo>
cd <your-repo>

# 2. Copy every file from this package into the repo, preserving structure:
cp -r {this-package}/* .
cp -r {this-package}/.claude .

# 3. Put the MASTER_PLAN.md from the previous deliverable at docs/MASTER_PLAN.md:
cp {previous-deliverable}/MASTER_PLAN.md docs/MASTER_PLAN.md

# 4. Commit.
git add .
git commit -m "chore: kickoff — rules, skills, Phase 0 guide"
git push origin main
```

### Day 1 — Start Claude Code in the repo

```bash
# From your repo root:
claude
```

Claude Code automatically reads `CLAUDE.md` and discovers the skills under `.claude/skills/`. Your first prompt should be something like:

> *"Read `docs/PHASE_0_STARTER.md` and `CLAUDE.md`. We're starting Phase 0. Let's start with Monday of Week 1 — solution skeleton per §2 of MASTER_PLAN. Go."*

Claude will:
1. Read the Phase 0 starter guide.
2. Read the relevant section of the plan.
3. Ask you to confirm the eight decisions if you haven't filled them in yet.
4. Run the scaffolding commands from Monday of Week 1.
5. Commit at the end of each logical chunk.

### Ongoing — how the skills get triggered

- **When you say "let's create the Billing module" or "scaffold the Wallet module"** → `erp-scaffold-module` triggers → Claude produces all 21 files per the skill.
- **When Claude is about to write a service method** → `erp-write-service` triggers → the method follows the `BaseService.ExecuteAsync` pattern.
- **When you say "Phase X is done" or "let's close Phase X"** → `erp-phase-check` triggers → Claude walks the 12 universal checks + the phase's extras and reports status.

You can also invoke a skill explicitly:

> *"Use the `erp-scaffold-module` skill to create the Inventory module per §6.3."*

### The workflow I recommend

1. **One phase at a time.** Don't start Phase 1 until Phase 0 is closed via `erp-phase-check`.
2. **One module at a time within a phase.** Don't start Billing until Identity is done.
3. **One module = 5–8 PRs.** Not one giant PR. Each PR:
   - Adds 1–3 entities + their configuration
   - Adds one or two service methods with full tests
   - Adds the matching controller endpoint
   - Green build, green tests, green arch — merges
4. **Run the skills every time.** The templates don't evolve based on mood; they evolve based on the plan.
5. **Don't argue with arch tests.** If an arch test fails, the code is wrong. Fix the code, not the test.
6. **Commit `docs/MASTER_PLAN.md` updates when reality diverges from the plan.** The plan is living; the rules in CLAUDE.md are not.

## What to do if...

| Situation | What to do |
|---|---|
| Claude skips a step in the skill | Point it at the skill explicitly: *"Run the `erp-scaffold-module` skill in full — don't skip the tests section"* |
| An arch test fails on code you "think is fine" | The test is right. Read the rule. Fix the code. |
| A plan section is ambiguous | Write the ambiguity to `docs/open-questions.md`. Decide with the human. Update the plan. |
| A new requirement lands mid-phase | Do not scope-creep Phase 0. Add to the plan; schedule for the right phase. |
| Claude produces code that drifts from the plan | Paste the plan's § reference in your next prompt. The plan wins every time. |
| You're tempted to skip a test class because "it's obvious" | Don't. The 28th module will have the same "obvious" test, and you'll appreciate having all 28 green. |

## What this package does NOT include (yet)

These emerge as you work, not upfront:

- Module READMEs (generated by `erp-scaffold-module` when each module is created)
- Specific entity migrations (generated per module as you build)
- Azure Bicep / Terraform infrastructure-as-code (write during Week 3 Friday of Phase 0)
- Per-module `.sql` files for stored procedures (add as needed)
- The actual Angular components (generated per module as you build)

These are **intentionally deferred** because they depend on concrete decisions you make in Phase 0 (cloud provider, auth strategy, domain names).

## Questions to ask before you start

1. **Who is the human reviewer?** Even solo work benefits from a second pair of eyes. PRs without review become bug factories.
2. **Where does the rubber meet the road for production?** You need a real deployed staging by end of Phase 0 — where?
3. **Who is the Product Owner?** (That's you, for now.) Write down their email — it seeds the first platform account.
4. **What's the kill switch?** If the project drifts from the plan by Week 6, what's the revert strategy?

If all four have clean answers, go. Otherwise, answer them first.
